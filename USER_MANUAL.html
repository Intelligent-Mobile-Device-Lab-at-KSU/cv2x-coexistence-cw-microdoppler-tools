VTC Fall 2026 Workshop — Tools User Manual

Welcome. This manual walks you through every tool in the TOOLS_FOR_VTC_FALL_2026_WORKSHOP/ directory, from “I have never touched this repo” to “I have a published-quality micro-Doppler spectrogram on disk.” It is written to be followed cold by a student or collaborator who has never seen the code before.

What these tools do

Three complementary radio-frequency (RF) sensing pipelines, all implemented in Python on top of GNU Radio and UHD, targeting a USRP X310 at 5.9 GHz:

  1. Hopped-CW micro-Doppler sensing. A single continuous-wave (CW) tone hops rapidly among four frequencies. After de-hopping on the receiver, the tone lands at DC and a short-time Fourier transform (STFT) yields a micro-Doppler spectrogram of whatever is moving in front of the antenna (a rotating panel, a walking pedestrian, a passing vehicle). Lives in multihop_and_cv2x_over_the_air_spectrogram_px/.
  2. C-V2X over-the-air capture, replay, and CW virtual sensing. Capture a real 3GPP Rel-14 C-V2X sidelink transmission, optionally inject a narrow CW sensing tone between data subcarriers, replay it GPSDO/DFN-aligned so a commercial radio still decodes the C-V2X data, and simultaneously extract micro-Doppler from the CW return. Lives in multihop_and_cv2x_over_the_air_spectrogram_px/overtheair/.
  3. Narrowband multi-subcarrier CW transceiver. A comb of N parallel CW tones symmetric around DC, with a live QT FFT display of both TX and RX. Separate paper, simpler signal, good for classroom demos. Lives in narrowband_paper/.

A pure-Python simulator (multihop_and_cv2x_over_the_air_spectrogram_px/simulator/) reproduces the hopped-CW chain with no hardware, so you can validate changes, tune parameters, and regression-test before ever touching a USRP.

2. Hardware and software prerequisites

Hardware

  • USRP X310 with two SBX daughtercards (or a comparable dual-channel setup). Radio #0 / subdev A:0 handles TX at 30.72 MSps (the full 20 MHz C-V2X channel), radio #1 / subdev B:0 handles narrow-band RX at roughly 192 kSps for the over-the-air CW return.
  • GPSDO supplying 10 MHz reference, 1 PPS, and GNSS time to the X310. The C-V2X replay pipeline is not optional: both capture and replay need GPS-disciplined time.
  • A commercial C-V2X radio (an OBU or RSU under test) on the same RF channel, if you want to verify that injected CW tones do not break PHY decoding.

Software

  • radioconda (or equivalent) with GNU Radio ≥ 3.9.
  • UHD 4.x with RFNoC — required for the FPGA-DRAM replay path (cv2x_rfnoc_replay.py).
  • Python packages: numpy, scipy, matplotlib with a TkAgg-capable backend, and PyQt5 (for mc.py's QT FFT sink).

Verifying the USRP and GPSDO

uhd_usrp_probe --args "type=x300"

Confirm that the GPSDO locks (sensors report gps_locked: true and ref_locked: true) before running any --gpsdo capture or replay. If the probe fails, scripts that query GPSDO sensors will fail too.

Why every command mentions a serial number

You will see the string serial=33767A5,master_clock_rate=184.32e6 repeated throughout this manual and in ALL COMMANDS FOR PAPERS - ANNOTATED.txt. That is the serial of the specific X310 these commands were developed on, and 184.32 MHz is the master clock rate that lets the X310 cleanly decimate down to 30.72 MSps (exactly one-sixth of 184.32 MHz).

Replace the serial with your own X310's serial (uhd_find_devices will print it). Keep the master_clock_rate value unless you know you want a different one.

Tip On Windows, the shell path to radioconda's Python typically looks like C:\Users\billy\radioconda\python.exe. You can call it directly instead of relying on PATH: for example C:\Users\billy\radioconda\python.exe cwhop_tx.py .... This avoids conflicts with any other Python installation.

4. Core concepts (quick glossary)

CW (continuous wave)

A pure sinusoidal tone at a fixed frequency. Nothing is modulated onto it; its only job is to reflect off moving targets and come back Doppler-shifted. Compact in spectrum, easy to analyse.

Micro-Doppler

The small, fast Doppler modulations a reflected CW tone picks up from non-bulk motion (a pedestrian's swinging arms and legs, a rotating fan blade, a vehicle's wheels). Visible as sinusoidal “butterfly” patterns in a spectrogram whose centre line is the bulk Doppler.

STFT (short-time Fourier transform)

Slide a window across the time-domain signal, take an FFT per window, stack the magnitudes into an image. Three tuning knobs matter:

  • Window length — longer windows give finer Doppler resolution but blur rapid changes. Typical values here: 10–70 ms.
  • Overlap — how much consecutive windows share. Higher = smoother time axis, more CPU. Typical values: 50–95%.
  • Zero-pad factor — multiply FFT size beyond window length (interpolates the frequency axis). 4× is the common sweet spot.

De-hopping

The TX tone jumps between frequencies (e.g. 7.5, 15, 22.5, 30 kHz). On the RX side you multiply by exp(-j·2π·fk·t) per dwell — where t is global sample time, never reset per block — so every dwell collapses onto DC and can be treated as a single virtual CW tone. The global-time phase is the whole trick.

Phase-continuous NCO

The transmitter uses a numerically controlled oscillator whose phase accumulator never resets at a hop boundary. That way the received tone has no phase jumps to confuse the de-hopper, and test_phase_continuity.py has something reasonable to validate.

GPSDO

A GPS-disciplined oscillator. Gives every USRP on Earth the same view of UTC time and a shared 10 MHz reference. Two radios locked to GPSDO can start a transmission on the exact same sample boundary, anywhere in the world.

DFN (Direct Frame Number) and why C-V2X replay needs GPSDO

C-V2X sidelink PHY scrambling is seeded by the Direct Frame Number, which cycles 0–1023 every 10.24 seconds (1024 frames × 10 subframes × 1 ms). The DFN is derived from UTC. If you capture an IQ file at time T and replay it later at an arbitrary time, the DFN at the first replayed sample will no longer match the DFN baked into the scrambler of the captured symbols — and a commercial C-V2X receiver will refuse to decode it.

The fix: capture with --gpsdo, snap duration to an integer multiple of 10.24 s, record the DFN/subframe of the first sample into a JSON sidecar, and on replay wait until UTC time lines up with the captured DFN before transmitting. The scripts cv2x_capture_replay.py and cv2x_rfnoc_replay.py handle this automatically when --gpsdo is set.

SC-FDMA

Single-carrier frequency-division multiple access, the uplink PHY used by C-V2X sidelink. Relevant here because you can simulate co-channel interference from it in the hopped-CW simulator, and because the CW injection trick slots a tone precisely between SC-FDMA subcarriers.

Sidecar JSON

A small .json file written next to every .cf32 capture that records things the raw IQ cannot carry: sample rate, centre frequency, UTC start time, DFN, hop schedule, CW injection parameters, references to other sidecars. Every processing script reads the sidecar to know what it is looking at. See §12 — File & sidecar formats.

.cf32 files

Raw interleaved complex-64 IQ samples on disk: each sample is two little-endian 32-bit floats, I then Q. Open with numpy.fromfile(path, dtype=numpy.complex64). The scripts stream these; a 20 s capture at 30.72 MSps is roughly 5 GB.

Dwell taper envelope

Each hop dwell fades in and out through a small taper (raised cosine, hann, blackman, linear, or none) rather than switching on and off abruptly. Smoother envelopes reduce spectral splatter at the cost of throwing away a bit of the dwell time.

5. Workflow 0 — Start in the simulator (no hardware)

Before you burn USRP time, validate the processing chain in the simulator/ directory. Nothing here touches hardware: pure Python, pure NumPy, all I/O is in RAM.

Why bother A clean butterfly in the simulator proves the math is right. If a hardware capture later looks broken, you immediately know the bug is in capture / LO / GPSDO / gain staging, not in the algorithm. Without this step you cannot tell those two cases apart.

Runnable scripts vs. library modules

The folder mixes two kinds of Python files. Scripts you can run directly from the command line are listed below. The other files (generate_hopped_cw.py, generate_reference_cw.py, generate_scfdma_data.py, make_taper_envelope.py, walking_micro_doppler.py, dehop_and_stft.py, compute_quality_metric.py) are library modules: they provide functions and dataclasses that the runnable scripts import. Do not run them directly — nothing will happen.

All commands in this section assume you have cded into the folder:

cd multihop_and_cv2x_over_the_air_spectrogram_px/simulator

5.1 Suggested learning path

Run these in order to understand the pipeline bottom-up:

1 — test_taper_envelope.py

python test_taper_envelope.py

Plots the dwell amplitude envelope for several taper shapes (raised_cosine, hann, blackman, linear, none) so you can see how taper_pct and min_amp affect the ramp. No CLI args — edit the hardcoded parameters at the bottom of the file to try different shapes.

What you should see: a matplotlib window with envelope curves overlaid. Smooth ramp regions, flat top at 1.0, reaching min_amp at the guard edges.

2 — test_phase_continuity.py

python test_phase_continuity.py

Generates a short hopped-CW signal and verifies the instantaneous phase is continuous across every dwell boundary. Prove-out for the phase-accumulator NCO model.

What you should see: console line PASS — max phase discontinuity = X.XX rad with X near zero, plus an optional plot of unwrapped phase vs. time that looks like a piecewise-linear staircase with no vertical jumps.

What failure looks like: a large phase jump at a dwell boundary, or a FAIL in the console. Caused by a bug in the phase accumulator (it reset when it should not have).

3 — test_pedestrian_model.py

python test_pedestrian_model.py

Generates a pure-CW signal modulated by the walking_micro_doppler 5-scatterer model (torso + 2 arms + 2 legs) and plots the spectrogram. This is what a walking human looks like to a perfect continuous-CW sensor — the ground truth the hopped pipeline has to reproduce.

What you should see: the classic micro-Doppler butterfly — a steady torso band near v_bulk, arm sinusoids oscillating at arm_f_rot Hz, leg sinusoids roughly 1.5× the arm Doppler and anti-phase to the arms. Edit PedestrianParams in the script to change walking speed, RCS mix, or carrier frequency.

4 — test_dehop_pipeline.py most important

python test_dehop_pipeline.py

End-to-end validation:

  1. Generate a hopped-CW signal with a pedestrian micro-Doppler modulation.
  2. Generate a matching continuous-CW reference.
  3. De-hop the hopped signal.
  4. STFT both.
  5. Plot side-by-side and compute quality metrics.

What you should see: a two-panel figure where the reference spectrogram (left) and the de-hopped spectrogram (right) show the same butterfly, plus console output with correlation ≈ 0.9+, sidelobe ≤ −30 dB, and a low MSE.

Regression gate Run this script after every edit to generate_hopped_cw.py, dehop_and_stft.py, or make_taper_envelope.py. If quality drops or the spectrograms diverge, your change broke something — revert.

5 — cwhopping_explorer.py interactive tuning

python cwhopping_explorer.py

Full tabbed GUI with every parameter exposed as a slider:

  • Tab 1 — Hopping CW: dwell_ms, hop_freqs, taper_shape, taper_pct, min_amp, transition_mode, gap_ms / gap_jitter.
  • Tab 2 — SC-FDMA Data: enabled, data_activity, num_subch_max, data_power_dB, spectral_isolation_dB, colocated_tx, cw_inside_alloc.
  • Tab 3 — Rx / Sim: pedestrian params (v_bulk, arm/leg Doppler & rate, RCS mix), STFT (window_ms, window_type, overlap_pct, zero_pad), duration, SNR.

Moving a slider regenerates the signal, reruns de-hop + STFT, and updates both spectrograms and the quality score live. Use this to find a good parameter combination before committing it to ../cwhopping_config_POST.json for a hardware run.

5.2 Common simulator workflows

A. Validate a config change before a hardware run

  1. Open cwhopping_explorer.py, move sliders until the quality score is acceptable.
  2. Mirror those settings into ../cwhopping_config_POST.json (see §11).
  3. Run test_dehop_pipeline.py as a final automated check.
  4. Only then run ../cwhop_tx.py on the X310.

B. Debug a de-hop artifact seen in a hardware capture

  1. Reproduce the capture conditions in cwhopping_explorer.py (same dwell, taper, hop frequencies).
  2. If the artifact appears in simulation → it is a signal-design problem, fix in the config.
  3. If it does not appear in simulation → it is a hardware issue (TX/RX timing, gain, LO leakage, GPSDO drift, etc.).

C. Regression-test after editing a library module

After changing generate_hopped_cw.py, dehop_and_stft.py, or make_taper_envelope.py:

python test_phase_continuity.py   # sanity
python test_dehop_pipeline.py     # end-to-end quality

Both should PASS / produce a clean butterfly. If either regresses, revert.

D. Benchmark quality for a paper table

compute_quality_metric.compute_quality_metric() returns a QualityMetrics dataclass with correlation, peak_sidelobe, dynamic_range, mse_dB. Call it from your own script in a parameter sweep over dwell_ms, taper_pct, etc.

Where outputs land Most simulator scripts are interactive (pop up a matplotlib window) and do not write files to disk by default. To save a PNG, add plt.savefig('out.png', dpi=150) before plt.show() inside the script.

6. Workflow 1 — Hopped-CW on hardware

This is the canonical production pipeline (“Workflow D” in ALL COMMANDS FOR PAPERS - ANNOTATED.txt). A single CW tone hops among four frequencies every dwell; both TX and RX are scheduled to the same GPS time on the X310 so the capture is sample-accurate.

All commands assume you are in the parent folder:

cd multihop_and_cv2x_over_the_air_spectrogram_px

Do not use If you are looking at old notes that mention test_hopper_tx.py, test_hopper_process.py, or doppler_viewer.py, stop. Those scripts are archived and use a different sample rate (192 kSps) and sidecar format from the current pipeline. The current replacements are, respectively, cwhop_tx.py, cwhop_rx.py, and cwhop_viewer.py. See §13 for the full archive list.

6.1 — Step 1a: TX/RX capture (cwhop_tx.py)

Transmits a phase-continuous hopped CW tone and captures the RX simultaneously. Both TX and RX are GPS-scheduled to the same UTC instant.

Three ways to run it

a) Generate the TX file to disk only, no hardware:

python cwhop_tx.py --config cwhopping_config_POST.json \
    --output my_capture --duration 30 --generate-only

b) Hardware TX + simultaneous RX capture (short form):

python cwhop_tx.py --config cwhopping_config_POST.json \
    --output my_capture --duration 60 --gpsdo \
    --tx-gain 15 --rx-gain 30

c) Full form with center frequency and UHD device args (the most common production call):

python cwhop_tx.py --config cwhopping_config_POST.json \
    --center-freq 5.9e9 --tx-gain 20 --rx-gain 30 \
    --duration 20 --gpsdo \
    --args "serial=33767A5,master_clock_rate=184.32e6" \
    --output my_capture

Outputs

FileWhat it is
my_capture.cf32TX baseband, complex64, 30.72 MSps
my_capture_rx.cf32RX capture, complex64, 30.72 MSps (same length as TX)
my_capture_schedule.jsonPer-dwell hop schedule: start/end sample, start/end ms, hop frequency

6.2 — Step 1b: Sanity-check the raw TX (tx_sanity.py or iq_spectrogram.py)

Before wasting time on a multi-stage processing run, verify the TX file looks correct: the CW tone should jump between the four hop frequencies (7.5 / 15 / 22.5 / 30 kHz) as the dwells advance.

Hop-aware (recommended when you have a sidecar):

python tx_sanity.py --input my_capture.cf32 \
    --sidecar my_capture_schedule.json --zoom-hz 20000 --chunk-ms 5

Draws a raw spectrogram with a cyan overlay indicating where the schedule says the tone should be. The overlay should sit exactly on top of the visible tone. If it drifts off, the sidecar or the TX timing is wrong — fix before processing.

Generic (works on any .cf32 file, no sidecar required):

python iq_spectrogram.py my_capture.cf32 -r 30.72e6 --decimate 64 --fft 64 --db-range 40

This viewer is hop-agnostic — use it to look at any IQ file, including C-V2X captures and narrow-band RX captures from the other workflows.

6.3 — Step 1c: Interactive viewer (cwhop_viewer.py)

Best for exploratory analysis and parameter tuning. Builds a decimated cache (approx. 30 s for a 20 s capture) on first open, then lets you tune STFT parameters live.

Minimal invocation (sidecar and config auto-found from the filename):

python cwhop_viewer.py --input my_capture_rx.cf32

With explicit sidecar and config:

python cwhop_viewer.py --input my_capture_rx.cf32 \
    --sidecar my_capture_schedule.json \
    --config cwhopping_config_POST.json

Force a rebuild of the cache:

python cwhop_viewer.py --input my_capture_rx.cf32 --reprocess

Recommended starting settings Once the viewer opens, set: window 70 ms, overlap 80%, zero-pad 4×, window type hann, dynamic range 40 dB, Doppler axis ±500 Hz, stitch ON. These are the values that produced the published result.

6.5 — Processing-chain diagram

my_capture_rx.cf32   (complex64, 30.72 MSps)
        |
        v
  De-hop: multiply by exp(-j*2*pi*delta_f*t_global) per dwell
  (phase-continuous; t is GLOBAL sample time, never reset per block)
        |
        v
  Cascade FIR decimate 3072x: [4,4,4,4,4,3] stages
  Anti-alias cutoffs: 3.84 MHz -> 960 kHz -> 240 kHz -> 60 kHz -> 15 kHz -> 5 kHz
  Block boundaries: zero-padded margins (NOT raw signal -- avoids FIR ringing)
        |
        v
  10 kSps baseband   (all dwells at DC, ~80 KB per original second)
        |
        v
  STFT spectrogram   (70 ms hann window, 80% overlap, 4x zero-pad)
  Resolution: ~2.4 Hz/bin   |   Range: +/-500 Hz
        |
        v
  Micro-Doppler spectrogram  (Doppler Hz vs time s)

Key design decisions

  • Phase-continuous TX. cwhop_tx.py accumulates global phase across dwells with no reset, so the received tone has no phase jumps at hop transitions.
  • Global-time de-hopping. De-hopping uses t = n / Fs with global sample index n, matching the TX NCO exactly.
  • Zero-padded margins, not raw signal. Block-boundary FIR margins are zero-padded rather than reading adjacent raw (un-de-hopped) signal. The raw signal at 7.5–30 kHz passes through the first four decimation stages unattenuated (stage-4 cutoff 60 kHz >> 30 kHz), which caused severe ringing at every block boundary when raw margins were used.
  • TX/RX cache separation. cwhop_viewer.py uses splitext only for cache naming, so my_capture.cf32 and my_capture_rx.cf32 never share a cache file.
  • GPSDO timing. TX and RX are both scheduled to the same GPS-locked time inside cwhop_tx.py, ensuring sample-accurate alignment. This fixed a 7–23-dwell misalignment that caused ghost images in earlier captures.

7. Workflow 2 — C-V2X capture and replay

Useful on its own for basic record/playback of a real C-V2X sidelink transmission, or as the first step of Workflow 3 (CW virtual sensing). 

cd multihop_and_cv2x_over_the_air_spectrogram_px/overtheair

DFN — non-negotiable If you omit --gpsdo, the replayed DFN drifts off the captured scrambler seed and a commercial C-V2X receiver will silently fail to decode. See §4 — DFN. Duration is auto-snapped up to the next 10.24 s multiple so the scrambler cycle closes cleanly.

7.1 — Capture a real C-V2X transmission

python cv2x_capture_replay.py --capture --gpsdo --headless \
    --rx-freq 5.915e9 --rx-gain 30 \
    --rx-args "serial=33767A5,master_clock_rate=184.32e6" \
    --capture-file cv2x_iq.cf32 --duration 20 -r 30.72e6

Outputs

FileContents
cv2x_iq.cf32Raw IQ, complex64, 30.72 MSps. Duration auto-snapped up to the next 10.24 s multiple.
cv2x_iq.jsonSidecar: capture_time_utc, dfn, subframe, cycle_offset_ms, sample_rate, center_freq, duration_s.

7.2 — Replay via GNU Radio

python cv2x_capture_replay.py --replay --gpsdo --headless \
    --tx-freq 5.915e9 --tx-gain 15 \
    --tx-args "serial=33767A5,master_clock_rate=184.32e6" \
    --replay-file cv2x_iq.cf32 --loop -r 30.72e6

The script reads cv2x_iq.json automatically. TX waits until UTC time matches the captured DFN/subframe and then starts on that sample boundary — the scrambler is valid from the very first sample.

7.3 — Replay via FPGA DRAM (RFNoC, recommended for sustained replay)

The GNU Radio path can underflow when the host can't keep up with 30.72 MSps, and a single-sample underflow breaks the 1 ms subframe grid. The RFNoC Replay Block path stores the IQ in X310 DRAM and plays it from the FPGA — no host-side jitter.

Upload once (stays in DRAM until power cycle):

python cv2x_rfnoc_replay.py --upload \
    --file cv2x_iq.cf32 -r 30.72e6 \
    --args "serial=33767A5,master_clock_rate=184.32e6"

8. Workflow 3 — C-V2X + CW virtual sensing (full chain)

The complete micro-Doppler-via-C-V2X pipeline: piggyback a narrow CW sensing tone inside the spectrum of a real C-V2X transmission, replay it so that the commercial OBU/RSU still decodes C-V2X data, and simultaneously extract micro-Doppler from the CW return. Four steps. 

cd multihop_and_cv2x_over_the_air_spectrogram_px/overtheair

The inject step is easy to miss The original command sheet skipped straight from “capture” to “replay.” Between those two steps, you must run cv2x_cw_inject.py to produce the CW-injected file. Without it, the “full-chain” replay is just a plain C-V2X replay with no sensing tone.

8.1 — Step 1: Capture a real C-V2X transmission

Same as Workflow 2 step 7.1. Produces cv2x_iq.cf32 + cv2x_iq.json.

8.2 — Step 2: Inject a CW tone between LTE subcarriers (cv2x_cw_inject.py)

python cv2x_cw_inject.py --input cv2x_iq.cf32 --output cv2x_iq_cw.cf32 --plot

The script detects active subframes in the capture, runs an FFT to find occupied subcarriers, and inserts a CW tone at an offset of (k + 0.5) × 15 kHz. Because LTE subcarriers sit on a 15 kHz grid, that 0.5-bin offset places the tone exactly between two data subcarriers. The commercial receiver's FFT gridlines ignore it and still decode C-V2X data cleanly.

Outputs

FileContents
cv2x_iq_cw.cf32Same IQ stream as the input, plus the injected CW tone
cv2x_iq_cw.jsonAll fields from cv2x_iq.json, plus a cw_inject block with f_cw_hz, subcarrier_active_idx, subcarrier_fft_bin, inject_power_db, inject_amplitude, active_subframes[], num_active, num_total

8.3 — Step 3: Simultaneous TX replay + narrow-band RX (cv2x_cw_txrx.py)

python cv2x_cw_txrx.py \
    --tx-file cv2x_iq_cw.cf32 \
    --rx-file cv2x_rx_doppler.cf32 \
    --args "serial=33767A5,master_clock_rate=184.32e6" \
    --tx-freq 5.915e9 --tx-gain 15 \
    --rx-gain 30 --rx-rate 192e3 \
    --duration 60 --gpsdo --loop
  • TX daughtercard replays cv2x_iq_cw.cf32 at 30.72 MSps (full 20 MHz C-V2X channel), GPSDO/DFN-aligned.
  • RX daughtercard tunes to tx_freq + f_cw_hz + rx_offset and captures a narrow ~192 kSps band around the CW tone.
  • rx_offset defaults to 50 kHz so the CW tone does not land on DC — keeps LO leakage from contaminating the sensor.

Outputs

FileContents
cv2x_rx_doppler.cf32Narrow-band RX capture, complex64, 192 kSps

8.4 — Step 4: Extract micro-Doppler offline (cv2x_microdoppler_extract.py)

python cv2x_microdoppler_extract.py \
    --input cv2x_rx_doppler.cf32 \
    --sidecar cv2x_iq_cw.json \
    --fft-size 19200 --overlap 0.9 \
    --doppler-range 500 --plot --save

What it does, in order

  1. Coarse FFT to find the exact received CW peak, correcting for the LO offset between TX and RX daughtercards.
  2. Mixes the CW tone to DC.
  3. Calibrates out the static LO offset.
  4. STFT spectrogram (Doppler vs time).
  5. Phase tracking at the CW frequency for fine-grained Doppler.
  6. Subframe gating — only shows windows where C-V2X was actually transmitting, using active_subframes from the sidecar.

10. Live monitoring during an experiment (cv2x_microdoppler_cw2.py)

Real-time visual feedback while you move a target in front of the antenna — useful for aiming, positioning, or quick qualitative demos. This is a completely separate workflow from the offline hopped-CW pipeline: single static CW tone, no hopping, no de-hopping, no config file.

Three panels update in real time:

  1. TX baseband spectrum — confirms the CW tone is actually transmitting.
  2. RX averaged spectrum — Doppler bumps either side of the CW peak when something is moving.
  3. RX waterfall — scrolling spectrogram, Doppler arcs left/right of centre.

Key tuning

  • --avg-frames 10 for fast gestures (a swinging hand).
  • --avg-frames 50 for slow vehicle motion.
  • Pure-software mode with no SDR: python cv2x_microdoppler_cw2.py -s 200e3.

Live only This script does not write a capture file suitable for offline processing. Use it for aiming and qualitative demos — then switch to the hopped-CW pipeline (§6) or the CW-injection pipeline (§8) for publication-quality spectrograms.

Hopped-CW file pairing

14. Troubleshooting

14.1 — Commercial radio won't decode after replay

  • Verify --gpsdo was set on both capture and replay. Without it, DFN drifts.
  • Verify capture duration is an integer multiple of 10.24 s (check duration_s in the sidecar).
  • Check GPSDO lock: uhd_usrp_probe --args "type=x300". Look for gps_locked: true.
  • On the RFNoC path, watch for U characters on stdout — underflows mean DRAM upload isn't complete or sample rate is misconfigured.

14.2 — CW tone not visible in the RX

  • Confirm f_cw_hz in cv2x_iq_cw.json matches what you expect.
  • Verify the RX tuned to tx_freq + f_cw_hz + rx_offset — cv2x_cw_txrx.py does this automatically from the sidecar.
  • Check TX/RX gain balance — the CW can be buried in C-V2X data if TX gain is too low or RX gain too high.

14.3 — Micro-Doppler spectrogram looks like noise

  • Run cv2x_microdoppler_extract.py --plot with no moving target. You should see a clean DC line. If you do not, LO calibration failed or the CW tone detection missed.
  • Check --fft-size matches your expected Doppler resolution. At 192 kSps, an FFT of 19 200 gives 10 Hz/bin.

14.4 — GPSDO query fails at script start

Scripts search for query_gpsdo_sensors in standard radioconda paths. If your installation uses a different layout, edit _QUERY_GPSDO_PATHS at the top of cv2x_capture_replay.py or cv2x_rfnoc_replay.py.

14.5 — Cyan overlay drifts off the tone in tx_sanity.py

The schedule sidecar or the TX timing is wrong. Do not proceed to the full processing run — fix capture timing first. Most common cause is running cwhop_tx.py without --gpsdo.

14.6 — cwhop_viewer.py cache looks wrong

Pass --reprocess to rebuild the _10k.npz cache. Always do this after you edit any module or switch to a new config.

14.7 — Simulator shows a clean butterfly but hardware doesn't

The bug is not in the core de-hop math — the simulator exercises that exact algorithm and it passes. Look at hardware-side problems: GPSDO lock drift, TX/RX gain staging, LO leakage, antenna alignment, replay underflows.

14.8 — cwhop_rx.py produces a clean spectrogram but cwhop_viewer.py doesn't (or vice versa)

The bug is likely in the block-processing / decimation layer of cwhop_rx.py. The simulator's dehop_and_stft.py is the reference implementation — if it looks right there, the issue is in the memory-efficient hardware version of the same algorithm.
